# 本地 POC 文档与阶段闸门计划

## 目标与成功标准
1. 在本地完成 AI 工单系统 POC 验证，不包含云端部署实现。
2. 采用 5 个阶段闸门机制，上一阶段测试全部通过后才能进入下一阶段。
3. 验收成功标准如下：
   - 问答准确率：`>= 85%`
   - 高情绪预警（`score > 0.85`）触发率：`100%`
   - 工程师通知延迟：`<= 30s`
   - 三端路由可访问且首屏加载时间：`<= 2s`

## 范围与非范围
1. 范围内：
   - 本地 FastAPI 服务、FAQ+检索问答、情绪分析、WebSocket 通知、三端 Web 页面、测试脚本与报告文档。
2. 非范围：
   - Docker、Nginx、HTTPS、PostgreSQL、云端 API、云端压测与上线。

## 系统架构（本地）
1. 后端：FastAPI（REST + WebSocket）。
2. 知识库：ChromaDB（SQLite 后端）。
3. 情绪分析：本地 Hugging Face 模型。
4. 三端界面：静态 HTML/JS，由 FastAPI 统一托管。
5. 实时链路：FastAPI 内存事件总线广播到工程师端与看板端。

## 公共接口与文档约束
1. 阶段状态枚举固定：`Not Started | In Progress | Blocked | Ready for Gate | Passed | Failed`。
2. 测试用例 ID 固定：`P{阶段号}-T{序号}`，例如 `P3-T2`。
3. 测试登记表列固定：`Case ID | Objective | Command/Action | Expected | Actual | Result | Evidence`。
4. 证据路径固定规则：`artifacts/phase-{n}/...`。
5. UI 规范优先级：`docs/agent.md` 高于其他文档中的 UI 描述，冲突时以 `docs/agent.md` 为准。

## 阶段计划与闸门

### Phase 1：本地环境与骨架
目标：FastAPI 可启动，基础目录可用。

任务：
1. 初始化项目目录与基础依赖。
2. 提供健康检查接口 `/health`。
3. 建立基础运行与启动说明。

测试用例：
1. `P1-T1` `python --version`，期望 `3.10.x`。
2. `P1-T2` `python -c "import fastapi,chromadb,transformers,torch"`，期望无异常。
3. `P1-T3` 启动服务后访问 `/health`，期望 `200`。

进入条件：
1. 需求文档与目录规范已确认。

退出条件（闸门通过）：
1. `P1-T1`、`P1-T2`、`P1-T3` 全部 `Passed`。
2. 在 `docs/poc_progress.md` 完成证据记录。

### Phase 2：问答与情绪核心
目标：FAQ+检索混合问答与情绪预警可用。

任务：
1. 实现 FAQ 规则匹配与检索回退。
2. 接入本地情绪分析，输出 `label/score/is_alert`。
3. 支持问题输入与响应结构化返回（答案、置信度、优先级）。

测试用例：
1. `P2-T1` FAQ 命中集评估（100 基础问题），期望准确率 `>= 85%`。
2. `P2-T2` 情绪样本评估（50 条），期望高情绪（`score > 0.85`）触发率 `100%`。
3. `P2-T3` 复杂工单抽测（10 条），期望返回答案 + 置信度 + 优先级。

进入条件：
1. Phase 1 闸门通过。

退出条件（闸门通过）：
1. 准确率与触发率达到目标。
2. 无阻断级错误。
3. `docs/poc_progress.md` 中含测试证据与结果汇总。

### Phase 3：实时通知链路
目标：工程师提醒与看板实时数据流跑通。

任务：
1. 建立 WebSocket 频道（工程师端、看板端）。
2. 告警事件与状态事件统一广播。
3. 记录事件发送与接收时间用于延迟评估。

测试用例：
1. `P3-T1` WebSocket 连接稳定性，期望持续连接无异常断开。
2. `P3-T2` 告警推送延迟测试，期望 `<= 30s`。
3. `P3-T3` 工单状态更新广播，期望工程师端与看板端同步变更。

进入条件：
1. Phase 2 闸门通过。

退出条件（闸门通过）：
1. 延迟达标，广播一致性通过。
2. `docs/poc_progress.md` 中记录事件日志与截图/日志证据路径。

### Phase 4：三端 UI 可用性
目标：客户端、工程师端、看板端按统一规范可用。

任务：
1. 完成 `/client`、`/engineer`、`/dashboard` 三端页面。
2. 按 `docs/agent.md` 执行 UI 一致性实现。
3. 联调 API 与 WebSocket 展示完整流程。

测试用例：
1. `P4-T1` 三端路由可访问，期望 HTTP `200`。
2. `P4-T2` 首屏加载时间检查，期望每端 `<= 2s`。
3. `P4-T3` UI 规范一致性核对（按 `docs/agent.md` checklist），期望无 P0/P1 偏差。

进入条件：
1. Phase 3 闸门通过。

退出条件（闸门通过）：
1. 三端可访问、性能达标、UI 规范检查通过。
2. `docs/poc_progress.md` 中完成 UI 验证记录。

### Phase 5：本地验收与报告
目标：形成可审计交付包。

任务：
1. 执行关键路径回归。
2. 整理测试结果与证据。
3. 输出本地验收报告与阶段结论。

测试用例：
1. `P5-T1` 全量回归（P1-P4 关键用例）全部通过。
2. `P5-T2` `local_test_report.md` 指标与证据完整。
3. `P5-T3` 演示流程走查（提问 -> 告警 -> 工程师处理 -> 看板更新）完整闭环。

进入条件：
1. Phase 4 闸门通过。

退出条件（闸门通过）：
1. `P5-T1`、`P5-T2`、`P5-T3` 全部通过。
2. 本地 POC 验收结论明确且可复现。

## 实施顺序（必须遵守）
1. 创建并维护 `docs/poc_plan.md`（本文件）。
2. 创建并维护 `docs/poc_progress.md`（状态与证据）。
3. 创建并维护 `docs/agent.md`（UI 唯一规范源）。
4. 每完成一个阶段，先在 `docs/poc_progress.md` 记录证据，再执行闸门判定。
5. 判定 `Failed` 时禁止进入下一阶段，只能修复并重测。

## 风险、依赖、回滚
1. 风险：
   - 模型推理耗时导致响应慢或通知延迟超标。
   - FAQ 与 KB 质量不足导致准确率不达标。
   - 三端 UI 不一致导致 P4-T3 失败。
2. 依赖：
   - Python 3.10 环境。
   - 本地模型下载能力与磁盘空间。
   - 浏览器和本地端口可用性。
3. 回滚策略：
   - 闸门失败立即停止推进，定位失败案例并形成修复单。
   - 修复后仅回归受影响阶段与上游关键用例，再重新判定闸门。
   - 任一阶段连续 2 次失败时，召开评审更新范围与标准。

## 默认假设
1. 文档语言使用中文，技术关键词保留英文。
2. 当前只覆盖本地 POC 阶段，不扩展云端章节。
3. 团队统一以 `docs/agent.md` 作为 UI 唯一规范源。
